# WeCom 技术通知消息最佳实践

本清单从一次备份同步通知的反复纠错 + 两个专项研究（notification best practices / alert best practices）中提炼。任何通过企业微信群机器人发出的技术状态/告警消息，发送前逐条核对。

## 1. 标题先行，先回答核心问题

第一句话必须说明发生了什么，不要埋结论。

```
Claude Code 备份同步完成 ✅
```

如果用户的核心问题是「会不会漏？」，标题或紧随其后的第一行就要给出答案，而不是放到末尾。

## 2. 区分「状态」与「增量」

- **状态（state）**：当前总量。例如：电脑上 1075 条主 session，备份里 2328 条。
- **增量（delta）**：本次实际变更。例如：今天 03:00 后新增 13 条主 session + 230 个其他文件。

不要说「今天同步了 1075 条」，除非 1075 条全部是今天新增的。

## 3. 每个数字必须带定义

```
- 源目录主 session：1075 个
- 备份主 session：2328 个
```

避免单独写一个「session: 1367」——读者分不清这是主对话、子代理输出、工作流日志还是全部文件。

## 4. 用具体地点，不用抽象词

| ✅ 推荐 | ❌ 避免 |
|--------|--------|
| 电脑上 / 源目录 | 当前 |
| 备份里 / 备份仓库 | 备份仓库（未解释时） |
| 过去保留的旧对话 | 含历史 / 旧 session（未解释时） |

## 5. 去掉噪音

默认不要包含：
- commit hash
- 文件绝对路径
- 内部文档引用
- 与当前问题无关的技术细节

除非用户主动要求审计信息，或这些细节能直接导向下一步动作（如 runbook/dashboard 链接）。

## 6. 技术用语要精确，不要「大白话」

宁可给一个带定义的技术术语，也不要为了「让人看懂」而模糊化。

```
✅ 主 session：用户与 Claude Code 的直接对话文件
❌ 就是那些聊天啦
```

## 7. 直接回答用户的核心担忧

如果用户问「会不会漏？」，消息里必须显式写：

```
- 验证：0 缺失，0 滞后
```

## 8. 给出下一步，或明确「无需操作」

结尾说明是否需要人工操作：

```
- 下一步：无需操作，下次自动同步 03:00
```

对于「完成」类通知，明确写「无需操作」能让读者立刻判断是否与自己有关。

## 9. 告警 vs 状态更新：不要混用同一套格式

| | 告警（Alert） | 状态更新（Status Update） |
|---|---|---|
| 目的 | 要求立即行动 | 告知一切正常 |
| 内容 | 症状 + 影响 + 严重程度 + 时间线 + 下一步 |  headline + 当前状态 + 变更 + 下一步 |
| 频率 | 只在需要行动时发 | 按计划发 |
| 结尾 | 「需要处理：是」+ 建议动作 | 「无需操作」 |

## 10. 告警消息的额外规则

### 10.1 可行动性第一

如果收件人看完消息后无法采取任何具体行动，就不要作为告警发送。告警必须 answer "What should I do first?"

### 10.2 报症状，不报原因

```
✅ API 错误率 > 10% 持续 5 分钟
❌ CPU 使用率 99%
```

只有当前因能直接推导出影响时间时，才允许附带原因。

### 10.3 必须包含的字段

- **影响**：谁/什么受影响，程度如何
- **严重程度**：P1–P5 或类似等级，保持一致
- **时间线**：何时开始、关键里程碑、预计下次更新时间
- **下一步**：正在做什么，或收件人应该做什么
- **相关链接**：dashboard、runbook、日志（可选但推荐）

### 10.4 使用结构化生命周期

对于持续事件，使用统一状态：

```
Investigating → Identified → Monitoring → Resolved
```

### 10.5 避免告警疲劳

- 不要发「FYI」类告警（如「磁盘 60% 满」）
- 同一根因的多个告警合并成一条 incident 消息，不要连发 4+ 条
- 只对有意义的状态变化发「已恢复」，不要每次瞬态抖动都发
- 不确定根因时诚实说「正在调查」，不要猜测

### 10.6 标题要具体

```
✅ 结算服务降级 — 500 错误上升
❌ 出现问题
```

## 11. 常见错误模板

| 错误说法 | 问题 | 改法 |
|---------|------|------|
| 1367/1367 条已同步 | 像进度条，且没说明备份总量 | 源目录：1075 条，备份：2328 条 |
| 当前 1367 条，备份仓库 2620 条（含历史旧 session） | 术语未定义，读者要做分类学 | 电脑上 1075 条主 session 已全部备份。备份里还额外保留 1253 条旧对话。 |
| 全部已备份，没有遗漏 | 太简略，像敷衍 | 验证：0 缺失，0 滞后 |
| 今天同步了全量 | 与每日增量同步矛盾 | 本次增量：13 个主 session + 230 个其他文件 |
| 出现问题 | 标题模糊 | 具体组件 + 症状，如「API 延迟 > 500ms」 |
| CPU 99% | 报原因不报症状 | API 错误率上升 / 响应时间变慢 |

## 12. 发送前自检问题

- 这条消息回答了什么具体问题？
- 哪个数字是增量，哪个是总量？
- 每个术语（session、备份、当前）第一次出现时是否已定义？
- 这是告警还是状态更新？是否用了对应的格式？
- 收件人是否需要立即行动？如果不需要，是否明确写了「无需操作」？
- 是否有读者必须自己推断才能理解的地方？
- 删除最后一段后，消息是否仍然完整？
- 手机上看是否超过 5 行？如果超过，是否每条都必要？
