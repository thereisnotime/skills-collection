# Task 02 — Org Auto-PoC Generation

**Project**: attacks-validation
**Trigger**: Cron 4×/day (06/12/18/00 UTC). Prompt-invoked when an analyst wants a specific queue row processed immediately.
**Skills**: `cve-poc-generator`, `script-generator`, `source-code-scanning`, `cve-risk-score`

## Inputs

- `OUTPUT_DIR`
- Optional `--scope-glob` to restrict to a specific queue row (event mode); default is `queue/scope-*.json` minus rows that already have a PoC.

## Procedure

1. Glob `$OUTPUT_DIR/queue/scope-*.json` (filter by `--scope-glob` if provided).
2. For each row where `$OUTPUT_DIR/assets/{asset}/pocs/{cve}.py` does NOT exist:
   1. Run `cve-poc-generator` skill on the CVE — produces a `poc.py` and a `report.md` in `$OUTPUT_DIR/artifacts/cve-pocs/{cve}/`.
   2. If the asset has source-code under `$OUTPUT_DIR/source/{asset}/`, run `source-code-scanning` to confirm the vulnerable code path is present and to bind the PoC to the right endpoint. If the affected sink is absent, mark the scope row `status: not_applicable` and continue.
   3. Run `script-generator` over the produced `poc.py` for syntax validation. If validation fails, regenerate once; on second failure, mark `status: poc_generation_failed`.
   4. Copy the validated PoC to `$OUTPUT_DIR/assets/{asset}/pocs/{cve}.py`.
3. Concurrency: max 4 PoC generations in parallel.
4. Emit status JSON with counts.

## Outputs

- `$OUTPUT_DIR/artifacts/cve-pocs/{cve}/poc.py` + `report.md` — canonical CVE artifact.
- `$OUTPUT_DIR/assets/{asset}/pocs/{cve}.py` — per-asset binding.
- Queue row `status` updated.

## Constraints — demonstrate, never disrupt

PoCs generated by this task are read by task-03 (live execution) and task-04 (weekly re-run). Every generated PoC MUST satisfy:

1. **Demonstrate, do not exploit.** Confirm the vulnerability exists by reading a minimal proof signal — a leaked env var, a single `/etc/passwd` line, an authenticated-as-other-user response, a returned secret. The PoC MUST NOT escalate beyond the proof.
2. **No state mutation.** Forbidden actions: `DROP TABLE`, `DELETE FROM`, file writes / deletes outside `/tmp`, account creation, password resets, sending email/SMS, posting to webhooks, mass-record reads (>10 rows), uploading files beyond what's needed to demonstrate the path, killing processes, restarting services.
3. **No DoS / resource exhaustion.** No fork bombs, no slowloris, no zip bombs, no infinite-recursion payloads, no auth-brute-force loops. PoC completes in ≤ 60 seconds.
4. **Read-only by default.** If a PoC absolutely needs a mutating action to demonstrate (rare — e.g. a stored-XSS that requires writing once), it MUST be gated behind explicit `--confirm` per `cve-poc-generator` Rule 7 AND the mutation MUST be reversible (e.g. write to a uniquely-named test record the operator can clean up).
5. **Bounded blast radius.** PoC targets exactly the scope row's asset + endpoint. Lateral probing is out of scope for this stage; the stitcher infers lateral movement from confirmed findings, not from the PoC itself.
6. **Evidence is observation, not action.** What gets captured under `evidence/raw-source.txt` is the response body / headers / leaked content — never includes a payload that would mutate the target on replay. The validator re-runs the PoC; replay must remain non-destructive.

The PoC generator is NOT run against the live target during this task — generation is offline. Live execution happens in task-03 only.
