# Scope layers — ln-030-vps-bootstrap

Deployment shape: **one VPS = one `BOT_USER=agent-bot` (shared) = one Anthropic OAuth + one Codex login + one nvm/Node toolchain**, then **one project = one Telegram bot token = one set of systemd units = one `PROJECT_NAME`/`PROJECT_DIR`/`SERVICE_PREFIX`/`RELAY_HOOK_PORT`**. The skill is parameterized so re-running it on the same VPS for a second project just adds new project-scope artifacts; the shared `agent-bot` user is reused.

| Layer | Scope | What lives here |
|---|---|---|
| **Global per-VPS** (one install per machine) | shared by every project | apt packages: `curl`, `wget`, `git`, `jq`, `gpg`, `pipx`, `python3` (native npm build tooling only), `bubblewrap`, `unzip`, `tmux`; `gh` CLI; upstream `glab` CLI; system-wide `/usr/local/bin/agent-update`, `agent-update.service`, `agent-update.timer`, `/var/lib/agent-update/`, `/var/log/agent-update.log`. Steps 1+2+7d are idempotent — running them twice is a no-op/update-in-place. |
| **Per-`${BOT_USER}=agent-bot`** (ONE shared Linux user — owns ALL projects) | scoped to the user's `$HOME` | `~/.nvm/`, Node 24, `claude` + `codex` CLIs (npm global within nvm), Claude user-scope plugins from `levnikolaevich-skills-marketplace`, `~/.claude/` (`settings.json` containing `model`/`effortLevel`/`permissions`/`statusLine`/`enabledPlugins`/`extraKnownMarketplaces` ONLY — NO `hooks` key, NO `CLAUDE.md`), `~/.claude/.credentials.json` (one Anthropic OAuth shared across all projects on this VPS), `~/.claude/statusline.sh` (shared script), `~/.claude/commands/<project-prefix>-dispatch.md` (project-named slash-commands stored at user-scope but addressable per project — names don't collide), `~/.claude/projects/<encoded-cwd>/<sid>.jsonl` (Claude Code auto-isolates sessions per cwd), `~/.codex/` (`config.toml` with multiple `[projects."..."]` blocks, one per project; `auth.json` shared). |
| **Per-`${PROJECT_NAME}`** (state, config, logs) | scoped to project dir name | `/etc/${PROJECT_NAME}/secrets.env`, `/etc/${PROJECT_NAME}/github-app.pem` (only when `GIT_PROVIDER=github` AND App auth used), `/var/lib/${PROJECT_NAME}/relay.db`, `/var/lib/${PROJECT_NAME}/god-command.json`, `/var/lib/${PROJECT_NAME}/sessions-dir.path`, `/var/lib/${PROJECT_NAME}/last-god-error.json`, `/var/log/${PROJECT_NAME}-god.log`. |
| **Per-`${SERVICE_PREFIX}`** (binaries, units, tmux) | scoped to systemd/binary prefix | `/usr/local/bin/${SERVICE_PREFIX}-god`, `/usr/local/bin/${SERVICE_PREFIX}-mint-gh-token` (only when `GIT_PROVIDER=github`), `/opt/${SERVICE_PREFIX}-agent-skills` (default `AGENT_SKILLS_DIR`), `/opt/${SERVICE_PREFIX}-relay-bot`, systemd units `${SERVICE_PREFIX}-god.service`, `${SERVICE_PREFIX}-dispatch.timer`, `${SERVICE_PREFIX}-dispatch.service`, `${SERVICE_PREFIX}-relay-bot.service`, tmux session `${SERVICE_PREFIX}-god`. Keep it explicit; set equal to `${PROJECT_NAME}` unless you want a separate service/binary prefix. `agent-update.service/timer` is intentionally system-wide, not per-service-prefix. |
| **Per-`${PROJECT_DIR}`** (project-scope claude config) | scoped to working directory of the god-session | `${PROJECT_DIR}/.claude/CLAUDE.md` (rendered from `references/operator.CLAUDE.md` with this project's identity — required because user-scope CLAUDE.md MUST be empty under shared `BOT_USER` to prevent cross-project identity leakage), `${PROJECT_DIR}/.claude/settings.json` (rendered from `references/settings.hooks.fragment.json` with this project's `RELAY_HOOK_PORT` — required because user-scope hooks would cross-route between projects). Cloned project repo also lives here once the operator runs `git clone`. |
| **Per-Telegram-bot** | one bot token per project | `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`. Telegram allows only one polling session per token, so you cannot share a bot across projects. The HTTP listener at `127.0.0.1:${RELAY_HOOK_PORT}` is also per-relay-bot — if you run a second project on the same VPS, override `RELAY_HOOK_PORT` (default `9999`) for the second instance. |
| **Per-cwd (Claude-managed, automatic)** | `~/.claude/projects/<encoded-cwd>/` | Claude Code itself isolates session JSONLs by the cwd it was started in. For `cd /opt/${PROJECT_NAME}`, jsonls land under `~/.claude/projects/-opt-${PROJECT_NAME}/<uuid>.jsonl`. Resume safety: `--resume <id>` only works against an id created in the same cwd. Cross-project resume is impossible by design. The wrapper `god-session.sh` resolves session UUIDs ONLY from per-project state (`/var/lib/${PROJECT_NAME}/last-session.id` + `/var/lib/${PROJECT_NAME}/sessions-dir.path`); it never uses `claude --continue` (which would otherwise pick newest session globally). |

**Multi-project on one VPS**: re-run Steps 3–9 with new vars. Step 3 finds the existing `agent-bot` user and reuses it (no second `useradd`, no second OAuth). Steps 1+2 are skipped/no-op except `glab` may upgrade in place. Step 4 (nvm) is also no-op. Step 5 (CLI install) re-runs `npm i -g` but they're already installed. Steps 5b/5c reuse the existing skills marketplace clone (only updates Codex `[projects."..."]` block to trust the new `${PROJECT_DIR}`). Step 6 sanity-checks. Step 7 creates new project-scope artifacts (units, dirs, logs, project-scope `.claude/`). Step 7d refreshes the same system-wide `agent-update` unit. Step 7b reuses user-scope statusline + claude-usage-report; renders new project-scope `CLAUDE.md`. Step 7c installs new relay-bot service for the new port; renders new project-scope `settings.json` with project-port hooks. Step 9 seeds new operator-side `.env.local` with the new project's `VPS_*` keys.

**Single-VPS-multi-project gotchas:**

- HTTP port `9999` is the default for `${SERVICE_PREFIX}-relay-bot.service`. Second project must set `RELAY_HOOK_PORT=9998` (or similar) in its `secrets.env`; rendered hooks (project-scope `settings.json`) and dispatcher config must use the same value.
- `/etc/sudoers.d/` rules (if any added later) must be `${SERVICE_PREFIX}`-scoped, not user-scoped (since BOT_USER is now shared).
- `StateDirectory=${PROJECT_NAME}` in `${SERVICE_PREFIX}-god.service` creates `/var/lib/${PROJECT_NAME}` regardless of unit name, so two projects with same `PROJECT_NAME` would collide — keep `PROJECT_NAME` unique per project.
- Filesystem isolation between projects relies on policy (operator.CLAUDE.md scope rules), not Linux DAC. Each project's claude has OS-level read access to other projects under `/opt/`, `/etc/`, `/var/lib/`. Hard kernel-enforced isolation between projects would require a per-Linux-user-per-project model, which this skill does not provide.
