# Troubleshooting — ln-030-vps-bootstrap

Reactive lookup for issues that surface during or after install.

| Issue | Solution |
|---|---|
| `nvm install` says success but `node` not found in next command | The pipe `\| tail` ran `nvm install` in a subshell; PATH side-effect lost. Re-run without piping. |
| `sudo -i -u ${BOT_USER} which node` returns empty | `.profile` patch missing. Re-run step 4's `printf >> .profile` block. |
| `agent-update.service` fails with `cannot read /home/${BOT_USER}/.nvm/nvm.sh` | Step 4 did not install nvm for `${BOT_USER}` or `agent-update.sh` was rendered with the wrong `BOT_USER`. Fix nvm, re-render `/usr/local/bin/agent-update`, then `systemctl daemon-reload`. |
| `agent-update.service` fails during `claude update` or `npm i -g @openai/codex@latest` | Network/registry/upstream CLI failure. The script exits before restarting any `*-god.service`, so existing sessions keep running. Re-run `systemctl start agent-update.service` after upstream recovery. |
| `agent-update.service` fails with `selected plugin not found` | `AGENT_SKILLS_PLUGINS` names a plugin absent from `${AGENT_SKILLS_DIR}/.agents/plugins/marketplace.json`. Use `agile-workflow`, `all`, or an explicit comma-list from that manifest. |
| `agent-update.service` fails with multiple LevNikolaevich marketplace blocks in Codex config | `~/.codex/config.toml` has an unmanaged duplicate outside the `ln-030 managed LevNikolaevich marketplace` block. Remove the duplicate manually; do not symlink Claude plugin roots into Codex. |
| Claude plugin update fails for `levnikolaevich-skills-marketplace` | The marketplace was not installed in Step 5c or Claude auth expired. Run `claude plugin marketplace add levnikolaevich/claude-code-skills` as `${BOT_USER}`, verify `claude plugin list --json`, then re-run the update service. |
| `agent-update.timer` is missing from `systemctl list-timers` | Upload/render `agent-update.timer` and `agent-update.service`, run `systemctl daemon-reload`, then `systemctl enable --now agent-update.timer`. |
| Nightly update ran while VPS was offline | Expected with `Persistent=true`: systemd catches up after boot/timer activation. Check `journalctl -u agent-update.service -n 100 --no-pager`. |
| `command contains null bytes or newlines` from `mcp__hex-ssh__remote-ssh` | Tool rejects literal `\n` in command string. Use `&&` chains or `printf '\n...'` for embedded newlines. |
| `claude /login` redirect to `localhost:8080` fails | SSH session was opened without `-L 8080:localhost:8080`. Reconnect with `ssh -i ${VPS_SSH_KEY} -L 8080:localhost:8080 -L 8081:localhost:8081 ${BOT_USER}@${VPS_HOST}`. |
| Modern paste-code login flow vs OAuth callback | CLI prints the URL — read what it says. Paste-code: enter long auth code shown on the auth-success page. OAuth callback: leave SSH `-L` open, browser redirect lands via tunnel. |
| Co-tenant container OOM during agent activity | RAM is tight. Either lower `MemoryMax` in `god-session.service`, or move `${BOT_USER}` workload to a dedicated small VPS (~€5/mo). |
| `gh auth login` says token has insufficient scopes | If using PAT instead of GH App: re-issue PAT with Contents R/W, Issues R/W, Pull requests R/W, Metadata R. Partial scopes silently break `gh issue edit --add-label`. |
| `tput: unknown terminal "unknown"` warning during scripted runs | Cosmetic. Caused by no TTY. Ignore or `export TERM=dumb` in the wrapper. |
| `Codex could not find bubblewrap on PATH ... will use the vendored bubblewrap` | System `bwrap` is missing — Step 1's apt install didn't run on this snapshot. `apt-get install -y bubblewrap`. |
| `gh` reports "You are not logged into any GitHub hosts" despite `GH_TOKEN=val gh ...` | `sudo -i -u <user>` spawns a subshell where bash inline-assignment doesn't propagate. Use `sudo -u <user> bash -lc '...'` (without `-i`). The `-l` flag still loads `.profile`. |
| `claude-usage-report` says «Кэш ещё не заполнен» | God-session has not made an API call since last restart. Send any Telegram message (or `tmux send-keys -t ${SERVICE_PREFIX}-god "ping" Enter`) to trigger the first response — statusLine fires on the API tick and writes the cache. |
| `/api/oauth/usage` returns `OAuth authentication is currently not supported` | Wrong/missing `anthropic-beta: oauth-2025-04-20` header. The endpoint is undocumented anyway — prefer the official statusLine path in Step 7b. |
| `claude-usage-report` percentages don't match the UI panel on the laptop | Cache is stale (script prints `⚠️ Данные обновлены N мин назад`). Send any Telegram message; statusLine fires on the next API tick and refreshes the cache. |
| `god-session.service` crash-loops with exit code 1; wrapper log shows `Permission denied` writing to `/var/log/${PROJECT_NAME}-god.log` | Log file owner does not match systemd `User=${BOT_USER}`. Fix: `chown ${BOT_USER}:${BOT_USER} /var/log/${PROJECT_NAME}-god.log && systemctl restart ${SERVICE_PREFIX}-god.service`. |
| `god-session.service` exits 3 with `FATAL: cannot read /etc/${PROJECT_NAME}/secrets.env` despite the file having mode 0640 and correct file owner | The PARENT directory `/etc/${PROJECT_NAME}` group ownership doesn't match `${BOT_USER}`, blocking traversal. Fix: `chown root:${BOT_USER} /etc/${PROJECT_NAME} && chmod 0750 /etc/${PROJECT_NAME}`. Always chown the directory itself, not just its contents. |
| `relay-bot.service` fails with `SqliteError: SQLITE_READONLY` on first pragma write | DB file at `/var/lib/${PROJECT_NAME}/relay.db` was created by another user (typically an inspection command like `sqlite3 ... .tables` run as root before the service first started) and the service running as `${BOT_USER}` cannot write. Fix: `unlink /var/lib/${PROJECT_NAME}/relay.db && systemctl restart ${SERVICE_PREFIX}-relay-bot.service` — relay-bot will recreate the file as `${BOT_USER}`. For verification, always use `sudo -u ${BOT_USER} sqlite3 ...`. |
| Bot for one project responds with another project's data (PRs, dispatch history, file contents) | Under shared `BOT_USER`, `~/.claude/CLAUDE.md` exists at user-scope and claude reads it as identity for every project. Fix: `unlink /home/${BOT_USER}/.claude/CLAUDE.md`; render the operator CLAUDE.md to project-scope only at `${PROJECT_DIR}/.claude/CLAUDE.md` per project and restart god-session. |
| Hooks fire to wrong relay-bot port (one project's tmux delivers to another project's relay-bot) | `~/.claude/settings.json` has a `hooks` key at user-scope. Fix: `sudo -u ${BOT_USER} bash -lc 'jq "del(.hooks)" ~/.claude/settings.json > ~/.claude/settings.json.new && mv ~/.claude/settings.json.new ~/.claude/settings.json'`, then ensure each project has its own `${PROJECT_DIR}/.claude/settings.json` with project-port hooks. |
| First start of new god-session hangs / dispatch.timer fires but nothing happens | claude is on first-run UX prompt (theme picker, login method, OAuth code, or trust-this-folder); `--dangerously-skip-permissions` does NOT bypass these gates. Fix: SSH in as `${BOT_USER}`, `tmux attach -t ${SERVICE_PREFIX}-god`, press Enter through theme picker / login menu / paste OAuth code / trust folder, then detach `Ctrl-b d`. Future projects under the same `${BOT_USER}` only need the trust-folder step (OAuth is shared). |
