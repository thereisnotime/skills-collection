hex-graph-mcp

Identity-first layered code graph MCP server. Deterministic indexing via tree-sitter for JavaScript, TypeScript/TSX, Python, C#, and PHP.

$ npm i -g @levnikolaevich/hex-graph-mcp Copy

Grep finds text. hex-graph understands code. It indexes your codebase into a SQLite knowledge graph via tree-sitter AST parsing — functions, classes, methods, imports, call edges. Then answers structural questions in milliseconds that would take an agent dozens of file reads.

The graph stores relationships, not just locations. When you ask “who calls this function?”, it doesn’t grep — it walks edges.

When it matters

Real scenarios where a knowledge graph changes the game.

🤔 Understand an unfamiliar codebase

A new team member asks “what does UserService do?” An agent with grep reads the class definition, then searches for callers, then reads those files, then searches for their callers. Four separate tool calls, 38,611 characters of context. An agent with hex-graph calls get_symbol(name="UserService", file="src/services/user.ts") — one request returns the definition, incoming edges, outgoing edges, structural context, and confidence summary.

38,611 → 1,274 ch. 97% savings. 4 tool calls → 1.
💣 Check blast radius before refactoring

“What breaks if I change calculateTotal?” Grep finds direct references but misses transitive dependencies — the function that calls the function that calls calculateTotal. hex-graph’s trace_paths walks reverse call, reference, import, type, or mixed paths from one canonical symbol identity. It finds the complete chain instead of just the first hop.

hex-graph returns MORE data (-167%) because it finds what grep misses: the full dependency chain. Completeness, not inefficiency.
🔍 Find all entry points

“Who ultimately calls validateInput?” The agent needs to trace callers 3 levels deep. With grep: read the file, find callers, read those files, find their callers, read again — 12,716 characters across 4 tool calls. With hex-graph: trace_paths(qualified_name="...", path_kind="calls", direction="reverse", depth=3) — one call returns the full call chain, served from the graph.

12,716 → 477 ch. 96% savings. 4 tool calls → 1.
🏗 Get architecture overview

New to a 50-file project? An agent needs to build a mental map. With built-in tools, it reads every file — 389,523 characters across 50 Read calls. With hex-graph: get_architecture() returns the module matrix (which directories depend on which), dependency counts between modules, and hotspot functions (most connected symbols). One call, 2,941 characters.

389,523 → 2,941 ch. 99% savings. 50 tool calls → 1. The single most impactful scenario.
🔄 Keep the graph current

Call watch_project once and forget about it. Chokidar monitors the file system — every save triggers an incremental AST reparse of just that file. Deleted files CASCADE-clean their graph nodes and edges. No manual reindex, no stale data. The graph is always current.

Incremental updates stay available, but index cost and break-even numbers now live under diagnostics instead of the public benchmark headline.

15 Tools

Query, analyze, and maintain the knowledge graph.

ToolWhat it does
index_project Scan and index project into a knowledge graph. Idempotent, skips unchanged files
search_symbols Full-text search for candidate symbols. Identity-safe discovery step before semantic queries
get_symbol Identity-safe symbol context: definition, incoming edges, outgoing edges, siblings, and module owner
trace_paths Unified path traversal across calls, references, imports, type, flow, or mixed graph layers
find_references Find semantic usages of one symbol identity: imports, calls, reads, type refs, and reexports
find_implementations Find type-layer implementations and overrides: extends, implements, overrides
find_dataflows Deterministic flow summaries and targeted interprocedural flow paths
explain_resolution Explain how a selector resolved, which candidates were considered, and why one won
get_architecture Module structure, cross-module dependencies, and hotspot summary from the layered graph
watch_project File watcher for incremental graph updates. Reuses the same indexing pipeline as full indexing
find_hotspots Identify refactoring targets. Functions/methods ranked by risk = complexity × callers
find_unused_exports Find exported symbols with no proven incoming usage. Confidence-aware outside JS/TS
find_cycles Detect circular module dependencies via Tarjan’s SCC algorithm
get_module_metrics Calculate coupling metrics (Ca/Ce/Instability) per file
find_clones Detect exact, normalized, and near-miss clones with suppression and scope controls

Architecture

Built on SQLite and tree-sitter — fast indexing, instant queries, zero external services.

  • Storage: SQLite via better-sqlite3. Single .codegraph/index.db per project. FTS5 for fast symbol search.
  • Parsing: tree-sitter WASM with 6 configured grammars: JavaScript, TypeScript, TSX, Python, C#, PHP.
  • Indexer: shared layered pipeline: PURGE → SCAN → PARSE → MATERIALIZE nodes → RESOLVE symbol/module/type/flow edges.
  • Watcher: Chokidar file system events. Incremental reparse on change, CASCADE cleanup on delete.
